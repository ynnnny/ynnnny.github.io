在现代化的开发中，一个人可能同时开发多个项目，安装的项目越来越多，所随之安装的依赖包也越来越臃肿，而且有时候所安装的速度也很慢，甚至会安装失败。

所以我们要去了解一下包管理器，在前端比较主流的包管理器主要有三个：npm,pnpm和yarn

在了解包管理器之前，我们先了解一下包管理的一个难题：幽灵嵌套

幽灵嵌套问题通常发生在依赖之间存在复杂的版本要求时，比如：

包 A 依赖于包 B@1.0.0
包 B 依赖于包 C@2.0.0
另一个包 D 也依赖于 C@3.0.0
在传统的依赖管理中，可能会导致包 C 的不同版本被嵌套在不同的子依赖树中，从而在 node_modules 中形成不同路径的多层嵌套，导致路径非常深。这种情况一旦发生，如果包 A 和包 D 不同意版本要求，可能会导致不同版本的包 C 被分别安装在不同的路径下，出现路径冲突甚至依赖问题，这就是“幽灵嵌套”现象。

NPM

npm（Node Package Manager）是 Node.js 的包管理工具，用于管理 JavaScript 和 Node.js 项目的依赖关系。它帮助开发者下载、安装、更新和管理项目所需的各种库和工具，是使用最广泛的一种包管理器。

从npm v2 到npm v7+的升级过程中，从最初使用递归的方法处理依赖，造成高度嵌套的依赖树，到后来使用扁平化管理，一定程度上解决了依赖嵌套问题，但是出现了算法时间过长的问题，最后引入了package-lock.json机制，作用是锁定依赖结构，一定程度上保持了依赖的稳定性。

核心：

  • 采用扁平化依赖管理管理方式
  • 每个依赖包都会在 node_modules 中单独安装
  • 相同的依赖可能会被重复安装多次

特点：

  • 优点：
  1. Node.js 默认包管理器，使用最广泛，拥有强大的额社区支持
  2. 最早的包管理器，简单易上手，对初学者友好
  3. package-lock.json 保证依赖版本一致性
  • 缺点：
  1. 安装速度较慢，占用空间大

目录结构：

  • npm 项目的目录结构通常包括以下几个关键部分。它是通过 npm init 或手动创建的 package.json 文件来定义的。常见的结构如下：

    1.node_modules/

    • 这是 npm 安装的所有依赖包存放的地方。它包含了你的项目所需要的所有第三方包。
    • 这个文件夹通常是自动生成的，不需要手动修改，应该被添加到 .gitignore 文件中以避免上传到版本控制系统中。

    2.package.json

    • 这是一个非常重要的文件，描述了你的项目的基本信息，如项目名称、版本、依赖、脚本等。
    • 该文件包含了：
      • **name**：项目名称
      • **version**：项目版本
      • **description**：项目描述
      • **dependencies**：项目所需的生产依赖（如框架、库等）
      • **devDependencies**：项目的开发依赖（如测试工具、构建工具等）
      • **scripts**：自定义命令，可以通过 npm run 执行
      • **main**：主入口文件
      • **author**：项目作者
      • **license**：许可证

    3.package-lock.json

    • 这是由 npm 自动生成的文件，记录了每个依赖的确切版本。它确保项目中每个人的依赖版本一致。
    • 这个文件不应该手动修改，npm 会根据 package.json 自动更新它。

    4..npmrc

    • 这个文件用于配置 npm 的一些个性化设置，比如注册表源、代理设置、缓存目录等。
    • 通常用于私有仓库的配置，或者更改默认的 npm 行为。

    5.README.md（可选）

    • 一个常见的文件，通常包含项目的介绍、安装指南、使用说明等。
    • 对于开源项目，这是一个很重要的文件。

    6.src/ 或 lib/（通常由开发者根据项目结构自行决定）

    • 这些文件夹通常用于存放项目的源代码。
    • src/ 存放开发中的源代码，lib/ 则存放已编译的代码或公共库。

    7.dist/ 或 build/（可选）

    • 用于存放构建后的代码，通常是通过打包工具（如 Webpack）生成的文件。
    • 这通常是可以直接发布到生产环境的文件。

YARN

Yarn 是一个 JavaScript 包管理工具，它由 Facebook、Google、Exponent 和 Tilde 开发，旨在解决 npm 在性能、可重复性和安全性等方面的一些问题。Yarn 是 npm 的替代工具，它提供了类似的功能，但在某些方面进行了优化，使得包的安装和管理更加高效。

核心：

1.并行下载提升安装速度：
传统的 npm 安装方式是依次下载依赖，而 Yarn 可以同时下载多个依赖，称为“并行下载”。这种方式充分利用了网络带宽，显著减少安装依赖所需的时间，使得安装速度更快。并行下载尤其在大型项目中效果显著，能够有效降低整体安装时间。

2.缓存机制减少重复下载：
Yarn 内置了缓存机制，在首次安装依赖时会将其缓存到本地。之后再次安装这些依赖时，如果依赖版本没有改变，Yarn 会直接从本地缓存中读取，而不是重新下载。这样不仅节省了网络请求，还提升了安装速度，特别适合离线开发和持续集成场景。

3.yarn.lock 确保依赖版本一致性：
Yarn 使用 yarn.lock 文件记录每个依赖的具体版本和来源，确保团队所有成员在不同机器上安装时得到的依赖版本完全一致。这避免了“依赖地狱”问题，即由于版本不一致导致的错误或不兼容情况，从而提高了开发过程的稳定性。

4.更安全的依赖解析机制：
Yarn 在安装依赖时会校验每个包的完整性（如 SHA 校验），以确保包的内容没有被篡改。这种安全机制能在下载依赖时检测到潜在的包篡改或恶意代码的引入，增强了项目的安全性。相比于早期的 npm，Yarn 的这种依赖解析机制更加严谨。

5.Workspace 支持更好的 monorepo 管理：
Yarn 支持 Workspace 功能，允许在单个代码库（monorepo）中管理多个项目或包。这种管理方式可以将多个子项目的依赖集中管理、共享，减少重复依赖的安装。此外，Yarn 还能通过 Workspace 在多个包之间建立相互依赖关系，使 monorepo 项目的开发、构建和测试更加高效。

主要特点：

优点：

  1. 采用扁平化优先 + 符号链接（符号链接是一个特殊的文件,它包含对另一个件或目录的引用路径）的组合策略
  2. 相同版本的包会被提升并复用
  3. 不同版本通过符号链接保持正确的引用关系

缺点：

1.仍然存在幽灵依赖问题：
尽管 Yarn 已经在扁平化和依赖管理上做了优化，但在一些复杂的项目中仍然会出现幽灵依赖问题。所谓幽灵依赖，指的是某个包在项目中使用但并未在 package.json 中声明，可能是通过其他依赖的间接依赖引入。这种隐式依赖会导致项目依赖关系难以维护，如果间接依赖被移除，可能会导致项目出错。

2.某些场景下的依赖解析较慢：
Yarn 的依赖解析虽然比传统 npm 更快，但在依赖结构复杂、依赖版本冲突较多的情况下，解析和处理依赖关系可能会变慢。尤其在 monorepo 中，Yarn 需要处理多个包之间的依赖关系，可能出现解析速度不如 pnpm 的情况。

与 npm 的比较：

  • 性能：Yarn 通过并行化和缓存提高了安装速度。
  • 一致性：Yarn 使用 yarn.lock 文件来锁定版本，确保每次安装时都使用相同的包版本，避免版本冲突。
  • 离线工作：Yarn 可以在离线环境中工作，只要包曾经被安装过。

PNPM

pnpm是一个快速、节省磁盘空间的 JavaScript 包管理器，它类似于 npm 和 Yarn，但提供了更高效的依赖管理机制。pnpm 通过独特的方式处理依赖包的安装，显著提高了安装速度，并减少了磁盘空间的使用。

核心：

1.采用内容寻址存储系统：
pnpm 使用内容寻址来存储依赖包。每个依赖包都会被哈希处理，并根据其内容生成唯一的存储地址。这样，即使多个项目依赖于相同版本的包，pnpm 也只需要存储一份，不会重复存储同样内容的文件。

2.使用硬链接和符号链接共享依赖：
pnpm 通过在 node_modules 中创建硬链接或符号链接（symlink），指向内容寻址存储中实际的依赖包。这样每个项目可以“共享”依赖，而不必为每个项目单独存储依赖包内容。

​ 1.硬链接（Hard Link） ：将文件内容链接到项目文件夹下，不占用额外磁盘空间。
​ 2.符号链接（Symlink） ：为特定版本的包创建路径映射，使项目代码能够准确找到每个依赖包版本的地址。

特点：
优点：
1.显著节省磁盘空间
2.安装速度快
3.更严格的依赖管理
4.pnpm-lock.yaml 确保依赖版本一致
缺点：
1.不兼容一些使用传统 node_modules 结构的工具和插件：

2.在 pnpm 中，每个依赖都有自己的隔离路径，某些工具、插件或构建系统可能会假设 node_modules 目录是扁平的，这可能导致兼容性问题。

3.与本地开发和测试环境的潜在不兼容：

4.有些项目依赖于本地 node_modules 结构，或者需要直接访问 node_modules 中的文件。在 pnpm 使用内容寻址和符号链接时，这可能会导致某些工具无法正常运行。

目录说明
.pnpm/ 文件夹：存放项目的所有依赖包，按 包名@版本号 命名，并在其 node_modules 文件夹中包含该包的实际文件和依赖项。

硬链接：.pnpm 中的实际文件并不是直接复制到每个项目中，而是通过硬链接指向 pnpm 的全局缓存存储目录 (pnpm store)。这样，不同项目间的相同版本依赖不需要重复下载。

符号链接：pnpm 会在项目的 node_modules 根目录创建符号链接，将每个包链接到 .pnpm 中实际的包路径。例如：

node_modules/react 是一个符号链接，指向 .pnpm/react@17.0.2/node_modules/react
node_modules/lodash 符号链接指向 .pnpm/lodash@4.17.21/node_modules/lodash

工作原理：

1.包安装：pnpm 会将依赖包下载到全局缓存 (pnpm store) 中，并将实际文件硬链接到 .pnpm 文件夹中的特定版本目录下。
2.创建符号链接：在项目的 node_modules 文件夹内创建符号链接，将包名称指向 .pnpm 中的对应路径。
3.引用：项目中的 require(‘react’) 会自动找到 node_modules/react 符号链接，并通过符号链接访问实际文件。

与 npm 和 Yarn 的比较

特性 npm Yarn PNPM
安装速度 较慢，重复下载依赖 比 npm 快，缓存机制更好 更快，使用硬链接和缓存
磁盘空间占用 高 相对较低 极低，利用硬链接节省空间
依赖管理方式 扁平结构，可能导致冲突 类似，但不严格限制 严格的依赖树结构
缓存机制 支持 支持，使用缓存加速 使用全局存储和硬链接
安装的效率 较低，尤其是相同依赖包多次安装 快于 npm，但不如 PNPM 极高，避免重复安装依赖
项目兼容性 完全兼容 完全兼容 完全兼容 npm 和 Yarn
使用选择：

基于这些特点：

  • 如果项目体积较小，团队成员 Node.js 经验不同，推荐使用 npm
  • 如果需要更好的性能和可靠性，推荐使用 yarn
  • 如果需要最严格的依赖管理、最小的磁盘空间占用，推荐使用 pnpm
